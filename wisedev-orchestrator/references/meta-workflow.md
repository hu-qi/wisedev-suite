# 元工作流控制面

本文件用于约束总控 Skill 的默认行为，作用类似“先检查技能、再进入具体任务”的元规则。

## 一、何时必须由总控 Skill 先介入

满足以下任一条件时，必须先由 `wisedev-orchestrator` 处理，再决定是否进入子 Skill：

1. 用户输入模糊，尚未明确当前阶段。
2. 用户输入可能跨越两个及以上阶段，例如“从会议纪要一直做到原型”。
3. 输入材料混合，既有纪要、又有规格片段、又有设计片段。
4. 用户直接要求下游产物，但上游产物明显缺失。

## 二、默认工作流

1. 判断输入起点。
2. 判断目标产物。
3. 判断当前阶段。
4. 生成上下文包。
5. 选择唯一子 Skill。
6. 输出当前阶段正式草稿。
7. 生成阶段交接单。

## 三、默认禁止事项

- 不要在阶段未判断清楚前直接输出大段结果。
- 不要同时混合两个子 Skill 的正式输出。
- 不要把 README 或人工审阅说明当成主要控制面。
- 不要在上游产物缺失时，静默脑补出完整下游产物。

## 四、样例调用优先级

### 复杂模糊输入
优先看：
- `../shared/full-chain-examples/case-01-upload-delivery/README.md`
- 当前阶段对应的完整链路文件

### 明确阶段任务
优先看：
- `references/example-library.md`
- 当前子 Skill 的 `references/examples/`

## 五、连续交付时的约束

当用户明确要求连续推进多个阶段时：

1. 先输出阶段计划。
2. 明确每一阶段的输入、输出和交接件。
3. 分阶段输出，不把所有阶段揉成单一大文档。
4. 在进入下一阶段前，显式复用上一阶段的追踪键与假设列表。
