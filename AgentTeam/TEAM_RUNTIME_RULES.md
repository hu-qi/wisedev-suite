# Team Runtime Rules

## 目的
本文件定义 WiseDev 长期协作团队在运行时的协作规则，用于约束 leader、成员和 reviewer 的实际行为。

目标是：
1. 保证共享工件始终是唯一正式事实来源
2. 保证阶段推进可控
3. 允许局部并行，但避免多人同时破坏一致性
4. 保证 reviewer 的审查真正发挥守门作用

## 一、共享工件规则

1. 所有正式结论必须写入共享工件。
2. 聊天消息、临时说明、口头推导不能替代共享工件。
3. 当共享工件与消息内容冲突时，以最新已验收共享工件为准。
4. 任何成员在推进任务前，必须先确认输入工件是否已验收。

## 二、Leader 派工规则

Leader 每次正式派工时，必须明确以下 5 项：
1. 负责人
2. 输入工件
3. 输出工件
4. 越界边界
5. 完成标准

如果以上任一项不明确，不应进入正式执行。

## 三、并行规则

### 允许并行的前提
只有当以下条件同时满足时，leader 才可允许局部并行：
1. 各任务输入边界清晰
2. 各任务输出工件不同或写入位置互不冲突
3. 并行任务之间没有未解决的强依赖
4. reviewer 或 leader 已确认当前上游事实足够稳定

### 默认允许并行的典型场景
1. 需求规格稳定后：
   - `solution_architect` 可启动设计
   - `api_designer` 可启动契约草稿
2. API 契约和设计约束稳定后：
   - `ux_designer` 可推进原型方向
   - `frontend_engineer` 可搭原型骨架

### 不允许并行的典型场景
1. 上下文包尚不稳定时，同时启动多个下游角色
2. 需求范围仍频繁变动时，同时输出设计与原型正式稿
3. reviewer 已指出高优先级冲突但尚未修复时继续下游开发

## 四、串行规则

以下阶段默认应严格串行：
1. 原始输入 -> 上下文包
2. 上下文包 -> 正式需求规格
3. reviewer 判定 fail 后 -> 任意下游推进

## 五、Reviewer 介入规则

### reviewer 必须介入的时机
1. 需求规格首次完成后
2. 设计文档首次完成后
3. OpenAPI 契约准备交给原型或前端前
4. 原型准备作为阶段交付物前
5. 任意关键工件发生重大变更后

### reviewer 输出要求
1. 必须引用具体工件
2. 必须说明影响范围
3. 必须给出推进建议
4. 不直接重写主产物

## 六、推进规则

### 可推进条件
只有在以下条件满足时，leader 才应允许进入下游阶段：
1. 当前阶段主工件已完成
2. reviewer 未给出阻断级问题，或阻断问题已修复
3. 关键假设、限制和待确认项已显式记录
4. 已生成阶段交接信息

### 不可推进条件
以下任一情况成立时，不应推进：
1. 输入工件仍存在核心冲突
2. 关键业务规则未明确
3. reviewer 判定为 fail
4. 角色越界输出导致共享工件失真

## 七、返工规则

1. 返工时必须明确返工负责人。
2. 返工时必须明确返工范围，避免全量重写。
3. 返工后必须重新提交 reviewer 审查。
4. 返工造成关键决策变化时，必须更新 `decision-log.md`。

## 八、决策记录规则

出现以下情形时，必须更新 `decision-log.md`：
1. 需求范围发生关键变化
2. 设计方案发生方向性调整
3. API 契约发生破坏性变更
4. 原型方向或交互范式发生重大调整
5. leader 明确裁定某项争议并形成正式结论

## 九、角色边界规则

1. `product_analyst` 不直接输出正式设计或实现代码。
2. `solution_architect` 不直接改写业务目标。
3. `api_designer` 不擅自创造核心业务规则。
4. `ux_designer` 不直接扩展需求范围。
5. `frontend_engineer` 不在无契约约束时重定义业务行为。
6. `reviewer` 不替代成员完成主产物编写。
7. `leader` 不长期吞并所有成员职责。

## 十、最小闭环规则

每一轮有效协作至少应包含以下闭环：
1. leader 确认主目标
2. leader 派工
3. 成员更新共享工件
4. reviewer 审查
5. leader 做接受 / 返工 / 推进决策
6. 必要时更新 handoff 与 decision log

## 十一、推荐执行策略

第一版运行时建议采用：
- 主线串行
- 局部并行
- reviewer 强介入
- leader 强验收

不建议第一版就采用：
- 全角色自由并行
- reviewer 弱化
- 仅靠聊天消息推进阶段
