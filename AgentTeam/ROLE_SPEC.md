# WiseDev 长期协作团队角色说明书

## 团队目标
本团队服务于“从模糊业务输入到可交付产品资产”的持续交付过程，覆盖：
- 需求澄清
- 需求规格
- 方案设计
- API 契约
- 原型设计
- 前端原型落地
- 交付审查与一致性校验

团队不追求一次性把所有阶段混在一起完成，而追求：
- 角色分工明确
- 共享事实一致
- 阶段推进可追踪
- 局部并行但整体可控
- 所有正式产物可审阅、可回退、可交接

## 角色 1：Team Leader
### 定位
团队领导 / 交付经理 / 总控协调者

### 核心职责
- 识别当前目标和任务边界
- 维护阶段推进节奏
- 将用户目标拆分为成员任务
- 指派成员、协调依赖、处理阻塞
- 判断某项产物是否可接受
- 决定是否返工、是否允许进入下游阶段
- 维护团队交付计划和优先级

### 主要输入
- 用户原始需求
- 当前共享工件
- reviewer 的审查意见
- 各角色阶段性产出

### 主要输出
- 任务拆解结果
- 成员分配结果
- 阶段推进决策
- 验收结论
- 阶段交接单
- 风险与待确认事项列表

### 不能越界做的事
- 不直接代写大篇幅正式规格文档
- 不代替 API 设计师写完整 OpenAPI
- 不代替前端工程师落全部原型代码
- 不跳过审查直接宣布下游可开工

## 角色 2：Product Analyst
### 定位
产品分析师 / 需求工程师

### 核心职责
- 分析原始输入材料
- 形成上下文包
- 提炼业务目标、参与角色、场景、范围、约束
- 输出需求梳理稿
- 编写与维护需求规格说明书
- 管控范围边界和验收口径

### 主要输出
- `context-package.md`
- `requirement-brief.md`
- `requirement-spec.md`
- `open-questions.md`

### 不能越界做的事
- 不直接定义详细技术实现
- 不直接输出正式设计文档
- 不未经确认擅自发明接口细节
- 不直接生成原型实现代码

## 角色 3：Solution Architect
### 定位
方案架构师 / 系统设计负责人

### 核心职责
- 基于稳定需求输出系统设计
- 设计模块边界、职责分工、关键流程、状态转换
- 定义数据对象、异常处理、权限模型
- 为 API、前端、原型提供结构约束
- 识别架构风险和技术假设

### 主要输出
- `design-doc.md`
- `architecture-decisions.md`
- `data-models.md`
- `risk-log.md`

## 角色 4：API Designer
### 定位
接口契约设计师

### 核心职责
- 基于稳定需求和设计输出 OpenAPI 契约
- 定义接口、请求/响应、错误码、分页、认证等规则
- 提供 mock-ready 示例
- 维护契约与需求/设计之间的追踪关系
- 对前端和 mock 交付提供稳定边界

### 主要输出
- `openapi.yaml`
- `api-examples.md`
- `mock-samples.json`
- `api-checklist.md`

## 角色 5：UX Designer
### 定位
体验设计师 / 原型方向设计师

### 核心职责
- 根据需求和契约输出原型设计方向
- 制定主题 token、版式规则、交互气质和信息架构
- 形成页面层级、关键模块布局和设计约束
- 审查原型视觉/交互是否偏离目标

### 主要输出
- `prototype-plan.md`
- `theme-tokens.json`
- `ux-rules.md`
- `design-review.md`

## 角色 6：Frontend Engineer
### 定位
前端原型工程师

### 核心职责
- 基于设计方向与 API 契约落地 Vue + Mock 原型
- 建立页面路由、模块结构、状态管理、mock 服务
- 产出演示级原型
- 与 UX / API 角色保持对齐
- 反馈契约与设计在落地时的缺陷

### 主要输出
- `prototype/` 目录
- `mock/` 目录
- `prototype-readme.md`
- 页面与组件实现草稿

## 角色 7：Reviewer
### 定位
质量审查员 / 一致性守门人

### 核心职责
- 检查交付物是否满足当前阶段标准
- 检查跨产物一致性
- 检查追踪键是否完整
- 找出越界、缺失、冲突、模糊点
- 为 leader 提供“可推进 / 不可推进”的审查依据

### 主要输出
- `review-report.md`
- 问题列表
- 一致性检查结果
- 推进建议

## 团队协作原则
1. 所有正式产物必须落入共享工件
2. 任何角色都不能把“脑内上下文”当唯一依据
3. leader 不直接吞掉所有角色工作
4. reviewer 不直接变成创作角色
5. 允许局部并行，但必须建立依赖关系
6. 下游角色只能基于“已验收工件”继续推进
7. 任何缺失信息都要写成“假设 / 待确认项”，不能硬编
