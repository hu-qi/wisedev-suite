# WiseDev Skills 套件使用说明与推荐提示词清单

## 1. 文档目的

本文用于系统梳理 WiseDev Skills 套件的两部分核心内容：

1. **安装与使用顺序建议**
2. **8 个 Skill 的推荐提示词清单**

目标是帮助团队在实际使用时做到：

- 明确每个 Skill 的定位与边界
- 知道模糊输入与明确输入分别该怎么走
- 知道多阶段任务与单阶段任务该怎么路由
- 快速复制可用提示词，稳定触发对应 Skill
- 在不同 AI 工具、多轮交互中尽量保持输出一致

---

## 2. WiseDev Skills 套件组成

当前 WiseDev Skills 套件共包含 8 个 Skill：

1. `using-wisedev`
2. `wisedev-orchestrator`
3. `wisedev-meeting-to-spec`
4. `wisedev-requirement-spec`
5. `wisedev-design-doc`
6. `wisedev-openapi-contract`
7. `wisedev-prototype-design`
8. `wisedev-vue-mock-prototype`

它们的关系不是并列平铺，而是一个分层结构：

- `using-wisedev`：元 Skill，负责先判断要不要进入 WiseDev 体系，以及应进入总控还是单阶段 Skill
- `wisedev-orchestrator`：总控 Skill，负责多阶段流程控制、上下文包、阶段判断、阶段交接
- 其余 6 个为阶段型子 Skill，分别负责需求梳理、需求规格、设计文档、API 契约、原型设计方向、Vue 原型

---

## 3. 推荐安装顺序

推荐以 **bundle 安装** 的方式整体安装 `wisedev-suite`。

### 推荐安装内容

安装时应完整保留：

1. `using-wisedev`
2. `wisedev-orchestrator`
3. `wisedev-meeting-to-spec`
4. `wisedev-requirement-spec`
5. `wisedev-design-doc`
6. `wisedev-openapi-contract`
7. `wisedev-prototype-design`
8. `wisedev-vue-mock-prototype`
9. `shared/`

### 3.1 为什么要这样安装

原因很简单：

- 前两个 Skill 负责“先判断、先路由、先控流程”
- 后六个 Skill 才是具体阶段的专业输出能力
- `shared/` 提供跨阶段模板、检查清单和完整链路样例，是整套 bundle 的公共依赖

如果只拆装单个 Skill 目录，而不保留 `shared/` 与完整目录结构，常见问题包括：
- 无法读取 `shared/full-chain-examples/case-01-upload-delivery/README.md`
- 无法稳定复用完整链路样例
- `wisedev-prototype-design` 与 `wisedev-vue-mock-prototype` 的交接信息不完整

如果先装后五个、不装前两个，AI 工具在面对复杂任务时还容易直接跳到某个阶段，导致：

- 没有先做输入归一化
- 没有先做阶段判断
- 没有先做上下文包
- 直接生成下游产物，导致上游缺失

所以，**最稳的方式永远是按 bundle 保留整个 `wisedev-suite` 目录结构**。

## 4. 推荐使用顺序

WiseDev 的使用顺序，取决于用户输入的清晰程度。

### 4.1 场景一：输入很模糊

常见输入示例：

- “根据这份会议纪要实现需求梳理”
- “做一个上传下达模块”
- “把这些材料整理成需求到原型”
- “根据这些材料给我做出 API 和前端原型”

#### 推荐走法

`using-wisedev` → `wisedev-orchestrator` → 按阶段进入对应子 Skill

#### 推荐步骤

1. 先让 `using-wisedev` 判断任务是单阶段还是跨阶段
2. 若为跨阶段或模糊输入，则必须进入 `wisedev-orchestrator`
3. `wisedev-orchestrator` 输出：
   - 输入归一化
   - 目标产物识别
   - 当前阶段判断
   - 上下文包
   - 唯一子 Skill 路由建议
4. 再按阶段依次推进：
   - 需求梳理
   - 需求规格说明书
   - 设计文档
   - OpenAPI YAML
   - 原型设计方向 / theme package
   - Vue + mock 原型

#### 为什么不能一步到位

因为模糊输入通常存在：

- 边界不清
- 角色不清
- 场景不清
- 上游信息不完整

直接一步生成最终原型，通常会导致：

- 需求漂移
- 设计过度脑补
- API 契约不稳定
- 原型结构不可复用

所以，**模糊输入必须优先走总控流程**。

---

### 4.2 场景二：输入已经很明确，只做单阶段

常见输入示例：

- “根据这份会议纪要提炼需求”
- “根据这份规格书输出详细设计”
- “根据这个设计文档生成 openapi.yaml”
- “根据 openapi.yaml 做 Vue 原型”

#### 推荐走法

由 `using-wisedev` 判断后，直接进入对应子 Skill。

#### 显式入口映射关系

- **会议纪要 / 录音转写 / 讨论材料**
  → `wisedev-meeting-to-spec`

- **需求规格说明书 / SRS**
  → `wisedev-requirement-spec`

- **概要设计 / 详细设计**
  → `wisedev-design-doc`

- **OpenAPI / 接口契约**
  → `wisedev-openapi-contract`

- **设计方向 / 视觉风格 / 配色方案 / 原型 theme**
  → `wisedev-prototype-design`

- **Vue + mock 原型**
  → `wisedev-vue-mock-prototype`

#### 使用原则

如果用户的目标已经非常清楚，并且输入也足以支撑当前阶段，就不必强行回退到总控。

但是只要出现以下情况之一，仍建议回到 `wisedev-orchestrator`：

- 输入混合了多类材料
- 一个请求要求多个阶段产物
- 当前阶段缺少必要上游产物
- 用户目标表达模糊
- 当前请求同时命中多个阶段入口

---

### 4.3 场景三：用户想一步到位

常见输入示例：

- “从会议纪要一路做到原型”
- “从需求直接给我 API 和前端 demo”
- “直接根据这些材料做个完整方案”

#### 推荐走法

仍然推荐：

`using-wisedev` → `wisedev-orchestrator`

由总控拆成阶段性交付，而不是直接跳到下游。

#### 推荐阶段序列

1. 输入归一化
2. 需求梳理
3. 需求规格说明书
4. 设计文档
5. OpenAPI YAML
6. 原型设计方向 / theme package
7. Vue + mock 原型

#### 核心原则

即使用户说“一步到位”，真正稳定的方式仍然是：

- 一次只产出一个阶段的审阅版结果
- 当前阶段确认后再进入下一阶段

这种方式更符合 WiseDev 的设计目标：

- 强调阶段边界
- 强调可追溯
- 强调上游约束下游
- 强调产物逐层沉淀

---

## 5. 最推荐的实际使用方式

如果是第一次正式使用整套 WiseDev，我建议按下面这条链路跑：

1. 先上传会议纪要或录音转写
2. 让 `using-wisedev` 判断任务类型
3. 进入 `wisedev-orchestrator`
4. 先产出需求梳理稿
5. 确认后继续生成需求规格说明书
6. 再确认后继续生成设计文档
7. 再生成 `openapi.yaml`
8. 再生成原型设计方向与 theme package
9. 最后生成 Vue + mock 原型

### 5.1 这样做的好处

- 需求不容易漂移
- 设计不会脱离规格
- API 契约更稳定
- 原型视觉方向可显式审阅，而不是隐含在代码里
- 原型更像“基于契约生成”，而不是自由发挥
- 更适合多人评审和多轮迭代

---

## 6. 推荐审阅顺序

如果当前不是马上使用，而是要检查这套 Skills 的质量，建议按下面顺序审阅：

1. `using-wisedev`
2. `wisedev-orchestrator`
3. `shared/full-chain-examples/case-01-upload-delivery`
4. `wisedev-openapi-contract`
5. `wisedev-prototype-design`
6. `wisedev-vue-mock-prototype`

### 6.1 为什么这样看

因为这 5 个部分最能体现整套方法论是否成立：

- `using-wisedev`：有没有先判断、先激活、先分流
- `wisedev-orchestrator`：有没有强制阶段工作流
- `case-01`：完整链路是否跑得通
- `openapi-contract`：契约是否足够稳定
- `prototype-design`：视觉方向、theme package 和评审机制是否足够清晰
- `vue-mock-prototype`：原型是否真的受契约与样例约束

---

## 7. 一条最实用的总原则

### 尽量让每次请求只聚焦一个阶段

即便用户最终想从需求一路做到原型，也建议每次只让 AI 聚焦：

- 当前阶段是什么
- 当前阶段需要什么输入
- 当前阶段输出什么产物
- 下一阶段该接什么

这样做的收益非常大：

- 输出更稳
- 结构更清晰
- 阶段边界更明确
- 更接近可复用的工业化交付方式

# 8. 8 个 Skill 推荐提示词清单

下面给出每个 Skill 的推荐提示词。目标是：

- 更稳地触发对应 Skill
- 更好地约束阶段边界
- 更适配 WiseDev 的设计逻辑

## 8.1 `using-wisedev`

### 用途

先判断是否要进入 WiseDev 体系，以及该走总控还是单阶段 Skill。

### 推荐提示词 1

请先按 WiseDev 的方式判断这个任务属于单阶段还是跨阶段任务。如果是跨阶段任务，请不要直接生成最终结果，先进入总控流程。

### 推荐提示词 2

请使用 WiseDev 体系处理这个任务。先识别输入类型、目标产物、当前所处阶段，再决定进入总控还是具体子 Skill。

### 推荐提示词 3

我会给你一份材料，请先不要直接输出结果。先按 WiseDev 元 Skill 规则判断：这是需求梳理、规格、设计、API 契约还是 Vue 原型任务；如果涉及多个阶段，请转入总控流程。

### 推荐提示词 4

请先执行 WiseDev 的激活判断。若当前请求命中多个阶段，必须先走 orchestrator，不要跳过上游阶段直接生成下游产物。

# 10. 结论与建议

## 10.1 最重要的使用建议

真正稳定使用 WiseDev 的关键，不是“一口气让 AI 全部做完”，而是：

- 先判断阶段
- 让每个阶段只做本阶段的事
- 当前阶段产物确认后再进入下一阶段
- 始终保留上游产物对下游的约束

## 10.2 最推荐的默认策略

如果你不确定该怎么开始，那就默认：

1. 先用 `using-wisedev`
2. 让它决定是否进入 `wisedev-orchestrator`
3. 如果进入总控，就按阶段推进
4. 如果明确单阶段，就直接走对应子 Skill

## 10.3 最适合这套 Skills 的典型任务

- 根据会议纪要梳理需求
- 根据需求梳理稿生成正式 SRS
- 根据 SRS 输出设计文档
- 根据设计文档生成 `openapi.yaml`
- 根据 `openapi.yaml` 生成 Vue + mock 原型
- 从会议纪要一路推进到可演示原型
